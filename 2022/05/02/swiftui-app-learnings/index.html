Creating Cross Platform SwiftUI App

Last summer I was really in need of a native multi-tab SSH client app that works well on my iPad, so I created SSHaking. It started as a UIKit app with AppDelegate and UITableViews, but a few pages (e.g. Preferences) where implemented in SwiftUI. After multiple iterations through the year, it has now became a modern SwiftUI cross platform app that works well on iOS and macOS. In this post, I‚Äôd like to share a few learnings that I‚Äôve gained through this journey.

Adopting SwiftUI

The declarative SwiftUI is a completely different world for the imperative UIKit engineers. Instead of manually creating views, setting up properties, and carefully organize the view hierarchy and constraints, you simply declare WHAT the view should look like without explicitly specifying the control flow. If you would like to learn SwiftUI, my recommendation would be Apple‚Äôs Introducing SwiftUI tutorials. Very well designed, super easy to follow.

Besides all these fancy views, one important change, in my opinion, is that we can now easily ensure a strong and reactive MVVM pattern through SwiftUI. In my app, most views have @State, @StateObject and/or @ObservedObject that describe the state of the view, and updates to these view models can be automatically propagated to Views and triggers a UI update. Embrace New App Lifecycle

Farewell, AppDelegate and SceneDelegate! SwiftUI 2.0 has introduced a brand new app structure / life cycle that allows you to declare the app‚Äôs structure as you may do with views. The brand new app lifecycle not only helped me removed so much boilerplate code, but also provides a centralized place to handle app level behaviors like deeplink and continue processing user activity with Handoff. I don‚Äôt want to cover too much about it here as I think it‚Äôs easier to understand if you create a new project and play with it. But here‚Äôs a tip: you can always create an AppDelegate class and add @UIApplicationDelegateAdaptor(AppDelegate.self) var appDelegate to your App struct so that you can fill in the gap between the fancy shiny declarative API and the good old robust delegates. For me, I still have an AppDelegate class to handle didFinishLaunching and notification related stuff, but the number of lines got a lot reduced than the old way.

Leverage Preview

SwiftUI comes up with a ridiculously strong preview system, which allows me to test my views live during coding and verify all use cases. You must have used previews if you ever tried SwiftUI, but do you know that you can create a Group of previews with different parameters so that you can see it in different devices, color schemes and orientations? I‚Äôm posting my usage here: for some important views, I‚Äôm able to display the iPhone and iPad, light and dark, portrait and landscape previews altogether. It definitely speeds up my development, and also helps guarantee that you won‚Äôt hit unexpected UI issues on a device/scheme/orientation that you didn‚Äôt test.

Maintain your own Swift Package

If you have multiple apps like me, you must have quite a lot shared code, and maintaining your very own Swift package is a great way to organize your private libraries and keep them in sync with all your projects. In my own Swift package, I have:

  • UI components, e.g. cells with image and switch in the settings view that is shown in above screenshot. SwiftUI really encourages you to create nested view structures, so don‚Äôt hesitate if you feel some parts can be extracted as a reusable View.
  • Managers to handle common/general logic, e.g. in-app purchase, data persistence, iCloud synchronization, popovers and alerts, preferences storage (spoiler: it‚Äôs PropertyWrapper!), etc.
  • Random utils, like converting hex to Color, formatting Date, reading properties from Bundle, etc.

Comparing to CocoaPods, Swift package is more intuitive (no podspec), more familiar (Swift vs Ruby), supports mutiple formats (source code, binaries, XCFramework), and it‚Äôs available out-of-box in Xcode. I do occasionally encounter issues that require a ‚ÄúReset Packages Caches‚Äù, but overall it‚Äôs very smooth experience.

Consider Platform Differences

Thanks to SwiftUI and MacCatalyst, my iPhone app could run and layout well on iPad and Mac, but it didn‚Äôt feel at home on such platforms. iPad and Mac tends to have larger screens, and multi column design is usually a better practice. The biggest iteration that I‚Äôve made to my app was rewriting the whole navigation stack using SwiftUI, which, thanks to the nature of declarative, automatically allows my app to run as normal UINavigationViewController on iPhone (actually, the compact size class, e.g. if you put an iPad app in split screen) and run in two columns with a sidebar on iPad and Mac.

Another key difference between iPad/Mac and iPhone, besides the extra screen size, is that iPads/Macs are more likely to be connected with an external keyboard and pointer device (trackpad or mouse). A few things to consider in your mind:

  • Keyboard shortcuts: Users would expect normal shortcuts to work, like cmd + t to create new tab, cmd + w to close current window/tab, cmd + , to open preferences window, etc.
  • Keyboard navigation: Use arrow keys to navigate through the list or controls. This would also be helpful for a11y.
  • Pointer interactions: Users who connect iPad to a pointer device like trackpad or mouse would expect controls to support pointer interactions. If you have custom controls, consider enabling/implementing pointer interactions when it‚Äôs proper. It will help your user interacts with your app with a better precision.
  • Keyboard input assistant items: When connecting to a physical keyboard, iPadOS shows a newly designed popover toolbar which works unfortunately poorly with UIInputView. In this case, consider using UITextInputAssistantItem, which can be displayed inline with the toolbar and creates more space for user to enjoy your app. Below screenshot shows UIInputView vs UITextInputAssistantItem when an iPad connects to an external keyboard.

UIKit is Your Last Resort

SwiftUI is nice, but there are still so many things that it cannot do (yet). UIKit is definitely your last resort and it is not hard to embed UIKit views or view controllers inside SwiftUI. In my app, for example, the core multi-tab terminal view is implemented in UIKit and WebKit, as the feature that I need simply doesn‚Äôt exist in SwiftUI. With that being said, the idea UI = f(state) still inspires me a lot even when writing UIKit code, and I ended up using some Combine code to create a similar reactive MVVM pattern.

Another use case is for when you want to access some UIKit specific features. Since SwiftUI is (for now) powered by UIKit, you can always just access the backed UIKit views. For the MacCatalyst version of my app, I really wanted the sidebar to have the pretty blurry transparent effect just like other native macOS apps, so I directly accessed the backed UISplitViewController and set primaryBackgroundStyle to .sidebar. To achieve this, you can embed an empty UIViewController to your SwiftUI View using UIViewControllerRepresentable, and access the splitViewController property of your view controller, as it shares the same ancestor as SwiftUI views. SwiftUI-Introspect is an awesome library that provides such APIs so that you can access the hidden UIKit world behind SwiftUI.

Final Thoughts

If you want to start writing an app from scratch, SwiftUI is a good choice to make your development efficient and quickly deliver prototypes. However, due to the huge feature gap and lack of functionalities, the most complicated part of your app is very likely to require UIKit.

If you already have a UIKit app and want to introduce SwiftUI, go for it. Start with migrating those most native, most simple views, like the boring forms and the table view of settings. If you find it helps, you can choose to continue migrate the rest part, even the complete app life cycle.

If you have an iPhone app and want to bring it to iPad and Mac, go through design guidelines and WWDC sessions to understand platform differences. Use your app daily in an unfamiliar way (trackpad and keyboard instead of touch). Leverage new APIs and UI design patterns to make your app feel at home on all platforms.
